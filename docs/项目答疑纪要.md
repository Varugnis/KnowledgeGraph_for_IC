# 半导体知识图谱项目 — 答疑纪要

> 本文档整理课程学习与答辩准备中的常见问题，便于撰写报告与小组讨论。  
> 对应仓库：`KnowledgeGraph_for_IC`

---

## 一、系统运行：模拟模式与 DeepSeek API

### 1.1 两种模式分别是什么？

| 模式 | 含义 | 是否需要联网/付费 |
|------|------|------------------|
| **模拟模式**（`--simulate`） | 用预设规则模板生成“类大模型”回答，不调用真实 API | 否 |
| **API 模式** | 将图谱检索结果填入 Prompt，调用 DeepSeek 生成回答 | 是（需 API Key） |

### 1.2 为什么分两种？

作业说明允许：“若无法调用大模型 API，也可用规则模板模拟大模型增强流程”。因此系统设计为：

- **无 API**：仍可完整演示 GraphRAG 流程（检索 → 拼 Prompt → 输出），满足作业要求；
- **有 API**：可真实对比“无图谱直接问答”与“图谱增强问答”，效果更好，适合答辩展示。

### 1.3 分工说明

- **与模式无关、始终相同**：图谱构建、规则/路径推理、子图检索（`build_kg.py`、`reasoning.py`、`graph_retrieval.py`）。
- **仅问答环节不同**：`llm_qa.py` 中是否调用 `call_llm_api()`。

推荐：答辩前配置 `DEEPSEEK_API_KEY` 运行 `python -X utf8 run_all.py`，展示 1～2 个对比案例（见 `results/cases.md`）。

## 二、知识图谱构建流程（数据采集 → 存储）

### 2.1 整体流程

```
公开资料（官网、规格、报道）
    ↓
Schema 设计（8 类实体、16+ 类关系）
    ↓
结构化人工整理 → data/raw/entities.csv、triples.csv
    ↓
build_kg.py → NetworkX 有向多重图 + kg_data.json
    ↓
visualize.py → 交互式 HTML / 静态图
```

### 2.2 各环节说明

| 环节 | 本项目做法 | 对应文件 |
|------|------------|----------|
| 数据来源 | 公开技术资料，人工核对 | `data/raw/*.csv` |
| Schema | 公司、芯片、工艺、架构、材料、器件、应用、技术等 | 见 README、entities 的 `type` 字段 |
| 信息抽取 | **结构化人工整理**（非自动流水线） | `triples.csv` |
| 图谱存储 | NetworkX + JSON 持久化 | `data/processed/kg_data.json` |
| 可视化 | pyvis 交互图 | `results/kg_full.html` |

### 2.3 图谱规模（满足作业要求）

- 实体：110 个（≥100）
- 三元组：318 条（≥300）
- 实体类型：8 类；关系类型：16 类

## 三、信息抽取方式归类说明

### 3.1 “结构化人工抽取”属于哪一种？

作业列举的四种**自动化**抽取方式：规则、正则、传统 NLP、大模型。

**结论：严格说不属于其中任何一种**，应表述为：

- **人工知识工程 / 结构化人工录入（Manual Knowledge Engineering）**

作业表述为“**可以**使用”上述方法，并非强制；小规模、高准确领域图谱采用人工 + Schema 是常见且合规的做法。

### 3.2 若需在报告中对应四类

- 仅可说明：Schema 对实体类型、关系类型形成**领域约束**（类似本体规范），但**抽取动作本身由人工完成**。
- **勿**写成“采用规则抽取建图”，以免与 `reasoning.py` 中的**规则推理**混淆。

### 3.3 建图抽取 vs 规则推理（勿混淆）

| 阶段 | 内容 | 模块 |
|------|------|------|
| 建图 | 人工写入 CSV，导入图 | `build_kg.py` |
| 推理 | 在已有图上用 Horn 规则、BFS 路径推导新知识 | `reasoning.py` |

## 四、为何建图阶段未采用四种自动抽取方法？

| 方法 | 未采用的主要原因（本项目） |
|------|---------------------------|
| **规则 / 正则** | 半导体表述多样（“4nm”“N4”“台积电代工”等），规则难覆盖；300 条规模下人工更省时间 |
| **传统 NLP** | 领域实体与细粒度关系（如 `uses_process`）通用模型效果有限，需标注与调参，课程周期成本高 |
| **大模型抽取** | 易产生工艺/代工幻觉，且仍需人工校对；与“用 KG 约束 LLM”目标冲突 |
| **人工 + Schema** | 准确率高、可标置信度、可溯源、便于 Git 协作与答辩讲解 |

**LLM 在本项目中的定位**：用于 **GraphRAG 问答增强**（`llm_qa.py`），不作为建图主数据源。

**扩展方向（可写在“不足与展望”）**：半自动抽取（LLM/规则生成候选三元组）+ 人工审核 + 现有 `build_kg.py` 流水线。

## 五、知识推理方法

### 5.1 已实现方法（满足“至少一种”）

本项目实现**两种**，并辅以链接预测启发式：

| 方法 | 作用 | 代码位置 | 典型结果 |
|------|------|----------|----------|
| **规则推理** | 两跳 Horn 规则，知识补全 | `reasoning.py` → `INFERENCE_RULES` | 约 104 条新三元组 |
| **路径推理** | BFS 多跳路径，证据链发现 | `bfs_paths()`、`path_based_reasoning()` | 路径推理准确率约 85.7% |
| **链接预测（辅助）** | 共同邻居 + 最短路径 + Jaccard | `link_prediction_heuristic()` | 用于实验对比 |

### 5.2 规则示例（便于记忆）

若 `公司A --designs--> 芯片B` 且 `芯片B --uses_process--> 工艺C`，  
则可推断 `公司A --company_uses_process--> 工艺C`（置信度按规则衰减）。

### 5.3 学习建议

1. 在 `data/raw/triples.csv` 中找 2 条事实，手画 2 跳路径；  
2. 对照 `reasoning.py` 中一条规则阅读 `apply_rule()`；  
3. 运行 `python -X utf8 src/reasoning.py`，查看 `results/reasoning_results.json`。

## 六、知识图谱对大模型的增强（GraphRAG）

### 6.1 采用的方法

**GraphRAG（图谱检索增强生成）**，而非 Text2Cypher。流程：

```
用户问题 → 实体检索 → 子图扩展（约 2 跳）→ 三元组转文本
    → 填入 prompts/kg_augmented_prompt.txt → LLM 生成（或模拟模板）
```

### 6.2 增强效果从哪里查看？

| 材料 | 内容 |
|------|------|
| `results/cases.md` | 无 KG vs 有 KG 的定性对比（准确性、证据链、多跳） |
| `results/metrics.json` | 幻觉检测、检索 P/R、路径推理等指标 |
| `results/qa_results.json` | 10 个测试问题的双路回答 |
| `results/kg_full.html` | 可视化核对三元组是否存在 |

### 6.3 四个维度的体现

| 维度 | 表现 |
|------|------|
| 准确性 | 如 H100 工艺由“4nm 左右”细化为“TSMC N4” |
| 证据可追溯 | 回答附带 `(主体, 关系, 客体)` 列表 |
| 减少幻觉 | `evaluate.py` 用图谱验证声明；案例如 M3 Ultra 非 Samsung 3nm |
| 多跳推理 | 路径推理 + Prompt 要求写出推理链（如 A17 Pro vs 骁龙 8 Gen 3） |

## 七、工程协作与仓库说明（摘要）

- **`.gitignore`**：忽略运行产物（`data/processed/`、HTML/PNG、`qa_results.json` 等），保留源码与 `data/raw/`、`results/cases.md`。  
- **克隆后运行**：`pip install -r requirements.txt` → `python -X utf8 run_all.py --simulate`。  
- **推送 GitHub**：需在系统终端完成认证（`git push`）；Cursor 内置终端可能无法弹出登录窗口。

## 八、答辩可用一句话总结

本系统以**结构化人工整理**构建半导体领域知识图谱，采用**规则推理与路径推理**补全与解释知识，通过 **GraphRAG** 将图谱子图注入大模型 Prompt，在准确性、可追溯性、抗幻觉与多跳推理上优于无图谱的直接问答；大模型用于增强而非建图，以体现“图谱约束生成”的设计思想。
